TunnelForge Export/Import Flow Review

작성일: 2026-06-01

검토 범위: Rust DB Core dump.run/dump.import, Python wrapper src/exporters/rust_dump_exporter.py, Import UI 흐름

검토 목적: 최근 FK collation 오류처럼 과거의 땜빵식 개선 또는 흐름상 근본 결함이 남아 있는지 확인

중요 전제: 이 리포트는 최초 결함 검토와 2026-06-01 수습 구현 이후의 최종 검증 상태를 함께 기록한다.

Executive Summary

현재 Export/Import 흐름은 그대로 운영 신뢰하기 어렵다. 특히 “안전 재생성 Import”는 다중 chunk 테이블에서 병렬 LOAD DATA 워커가 임시 스키마가 아니라 원래 대상 DB로 접속하는 구조라, 안전 재생성이라는 이름과 다르게 원본 대상 DB에 쓰기 작업이 들어갈 수 있다.

또한 Export는 일관 스냅샷 없이 여러 커넥션/여러 시점에서 데이터를 읽고, Import는 strict SQL mode를 완화한 뒤 성공 시 row count/digest 검증 없이 성공을 반환한다. UI에서 제공하는 timezone 설정은 Python wrapper payload에 반영되지 않는다.

Flow Map

Export

  1. Python RustDumpExporter._run_rust_dump builds payload with data_format=tsv, compression=zstd, overwrite=true.
  2. Rust dump_run validates output dir, inspects schema, orders tables, estimates row counts, chooses sequential/table-parallel/global MySQL dump path.
  3. Chunks are written as TSV/JSONL, checksums are recorded, then _tunnelforge_dump.json and marker are written at the end.

Import

  1. Python RustDumpImporter.import_dump reads manifest for UI metadata, builds dump.import payload, and optionally passes selected retry tables.
  2. Rust dump_import reads manifest, validates paths/checksums when present, orders tables, and applies mode behavior.
  3. replace: drop selected/import tables, create tables, import data, create secondary indexes/FKs.
  4. merge: skip create/drop, import data into existing tables, then still attempts post-load DDL.
  5. recreate full MySQL: create shadow schema, create tables there, import data, apply post-load DDL, drop/recreate target DB, rename shadow tables into target DB.

Findings

CRITICAL Safe recreate 병렬 import가 원본 target DB로 접속한다

근거: safe recreate에서는 메인 adapter만 USE shadow_database 한다. 하지만 다중 chunk TSV import는 import_mysql_tsv_table_parallel(endpoint, ...)로 분기하고, 각 worker가 LiveAdapter::connect(&endpoint)로 새 접속을 만든다. 이 endpoint는 원래 target database를 가리킨다.

증거 위치: migration_core/src/lib.rs:3087-3121, migration_core/src/lib.rs:3260-3268, migration_core/src/lib.rs:3706-3714, migration_core/src/lib.rs:3819-3834

영향: “안전 재생성 Import”가 임시 스키마에만 쓰지 않는다. 다중 chunk 테이블은 기존 target DB에 LOAD DATA가 들어가거나, target에 테이블이 없으면 실패한다. 실패 후 switch 로직에 진입하지 않더라도 이미 target DB에 쓰기 부작용이 발생할 수 있다.

권장 조치: safe recreate 상태에서는 worker endpoint database를 shadow database로 복제해 넘기거나, parallel worker에 schema override를 명시 전달한다. 이 버그가 고쳐지고 회귀 테스트가 생길 때까지 safe recreate는 threads=1로 강제하거나 UI에서 비활성화해야 한다.

CRITICAL Export에 consistent snapshot/transaction 경계가 없다

근거: dump_run은 schema inspect, row count, chunk dump를 별도 단계와 여러 커넥션에서 수행한다. 코드 검색에서 START TRANSACTION, WITH CONSISTENT SNAPSHOT, LOCK TABLES, SET TRANSACTION 경로가 확인되지 않았다.

증거 위치: migration_core/src/lib.rs:1529-1655, migration_core/src/lib.rs:1916-1985, migration_core/src/lib.rs:2757-2998

영향: 운영 DB가 export 중 변경되면 table 간 FK 일관성, row count, chunk 내용이 서로 다른 시점의 데이터가 될 수 있다. Export 완료는 “덤프 파일이 만들어졌다”만 의미하고 “복원 가능한 일관 스냅샷”을 의미하지 않는다.

권장 조치: MySQL은 InnoDB 기준 START TRANSACTION WITH CONSISTENT SNAPSHOT READ ONLY 전략을 검토해야 한다. 다중 커넥션 병렬 dump를 유지하려면 snapshot token/backup lock/전역 read lock 등 설계가 필요하다. 최소한 UI와 manifest에 “consistent=false”를 명시하고 운영 Export 전 변경 중지를 요구해야 한다.

HIGH Merge/Retry import가 데이터 적재 후 기존 index/FK를 다시 생성한다

근거: merge 모드는 create/drop 단계를 건너뛰지만, 데이터 import 후 apply_post_load_ddl을 무조건 호출한다.

증거 위치: migration_core/src/lib.rs:3158-3202, UI retry는 src/ui/dialogs/db_dialogs.py:2701-2703에서 merge로 강제된다.

영향: retry/merge 대상 테이블에 이미 index/FK가 있으면 duplicate key/index/constraint 오류가 발생할 수 있다. 오류가 데이터 적재 뒤에 발생하면 일부 데이터만 들어간 뒤 실패하는 상태가 된다.

권장 조치: merge에서는 post-load DDL을 기본적으로 생략하거나, 대상에 없는 index/FK만 diff 기반으로 생성해야 한다. retry는 “실패 테이블 cleanup 후 replace”인지 “기존 테이블 append”인지 의미를 분리해야 한다.

HIGH UI timezone 설정이 Rust import payload로 전달되지 않는다

근거: UI는 timezone_sql을 계산해 worker로 전달하지만, Python wrapper의 payload에는 target, input_dir, mode, threads만 포함된다.

증거 위치: src/ui/dialogs/db_dialogs.py:2670-2712, src/exporters/rust_dump_exporter.py:594-647

영향: TIMESTAMP/DATETIME 계열 데이터에서 사용자가 선택한 KST/UTC 보정이 실제 import session에 적용되지 않는다. UI상 제어가 있는 것처럼 보이지만 core는 받지 않는다.

권장 조치: Rust core payload에 session_sql 또는 제한된 time_zone 필드를 추가하고, 모든 main/parallel import connection에 동일하게 적용해야 한다. 임의 SQL 주입 형태는 피해야 한다.

HIGH Import 성공 기준에 row count/digest 검증이 없다

근거: Import는 rows_imported, chunks_imported만 반환한다. manifest의 table.rows와 대상 table count 비교, digest 비교, FK 검증 결과를 success gate로 삼지 않는다.

증거 위치: migration_core/src/lib.rs:3190-3226

영향: SQL mode 완화, LOAD DATA coercion, 잘못된 기존 데이터, retry 중복, 부분 적재가 있어도 core가 “성공”으로 종료할 여지가 있다.

권장 조치: 최소 table별 target count == manifest rows 검증을 success gate로 추가한다. 중요 테이블은 canonical row digest 또는 checksum 기반 검증을 옵션으로 추가해야 한다.

HIGH UI는 “모든 객체 재생성”이라고 안내하지만 dump/import는 tables만 처리한다

근거: UI 문구는 “테이블/뷰/프로시저/이벤트” 재생성을 안내한다. Rust dump manifest는 NormalizedSchema.tables와 table chunks만 저장한다. unsupported objects는 inspect에서 수집되지만 dump_run에서 차단하거나 manifest에 보존하지 않는다.

증거 위치: src/ui/dialogs/db_dialogs.py:2065-2070, migration_core/src/lib.rs:1529-1668, migration_core/src/lib.rs:4275-4279

영향: Export/Import 성공 후에도 views/procedures/triggers/events가 복원되지 않을 수 있다. 운영 복구 도구로 쓰는 사용자에게 실제 DB 상태와 UI 설명이 다르다.

권장 조치: 당장 UI 문구를 “테이블 데이터/인덱스/FK 중심”으로 정정하고, unsupported objects 존재 시 Export 완료 전 경고/리포트로 남긴다. 장기적으로 객체별 DDL 캡처/복원 설계가 필요하다.

MEDIUM Checksum 검증이 optional이라 manifest에서 누락되면 무결성 검사가 통과한다

근거: validate_dump_manifest_chunks는 chunk 파일 존재를 확인하지만, chunk_sha256.get(...)이 None이면 checksum 비교 없이 넘어간다.

증거 위치: migration_core/src/lib.rs:7505-7535

영향: 새 format v2 dump에서 checksum이 누락되거나 손상되어도 import가 진행될 수 있다. 오래된 v1 호환 때문이라면 version별 정책이 분리되어야 한다.

권장 조치: format_version 2 이상은 모든 chunk checksum을 필수화한다. v1 호환은 명시적 legacy path로 분기하고 UI에 “무결성 검증 불가”를 표시한다.

MEDIUM SQL mode 완화가 데이터 변형을 숨길 수 있다

근거: import 시작 시 strict/zero-date 관련 mode를 제거하고 foreign_key_checks=0, unique_checks=0를 설정한다.

증거 위치: migration_core/src/lib.rs:3597-3609

영향: Import를 성공시키기 위해 coercion이 발생해도 후속 검증이 없으면 원본과 다른 값이 들어갈 수 있다.

권장 조치: 완화 모드는 사용자가 선택한 호환 모드에서만 켠다. 기본은 strict 유지, 실패 시 어떤 컬럼/값 때문에 실패했는지 리포트하는 방향이 맞다.

MEDIUM Table-level charset/collation/engine 보존은 아직 불완전하다

근거: 최근 수정으로 문자열 컬럼 charset/collation은 보존되지만, table default charset/collation과 engine option은 schema model에 별도 필드로 보존되지 않는다. MySQL DDL은 현재 ENGINE=InnoDB만 붙인다.

증거 위치: migration_core/src/lib.rs:4200-4240, migration_core/src/lib.rs:8709-8727

영향: FK 문자열 호환 문제는 새 export에서 개선되지만, 전체 schema fidelity는 아직 원본과 다를 수 있다. MyISAM, table default collation, row format 등은 손실된다.

권장 조치: NormalizedTable에 engine/default charset/default collation/options를 구조화해서 저장하고, DDL 생성에 반영한다. type string에 정보를 덧붙이는 방식은 임시 호환층으로만 사용한다.

MEDIUM 기존 dump에는 charset/collation이 없으며 자동 복구할 수 없다

근거: 사용자가 제공한 기존 dump manifest의 audit_category.code, df_evaluation_results.audit_category_code는 둘 다 varchar(45)만 보유했다. 원본 collation 정보가 dump에 없다.

영향: 기존 dump는 어떤 collation이 원본이었는지 알 수 없으므로, 새 코드만으로 원본 스키마를 정확히 복원할 수 없다.

권장 조치: 기존 dump import 전 preflight에서 “charset/collation metadata missing”을 탐지해 warning 또는 block한다. 원본 DB 접근이 가능하면 재-export를 요구하는 것이 정직한 동작이다.

Verified Positive Controls

  • OK Dump table path와 chunk path는 canonicalize 후 dump directory 하위인지 검사한다. 관련 코드: migration_core/src/lib.rs:7463-7503.
  • OK Post-load DDL은 index를 먼저 만들고 FK를 나중에 만든다. 관련 코드: migration_core/src/lib.rs:7856-7965.
  • OK Python metadata 분석도 manifest path traversal과 chunk symlink 외부 참조를 거부하는 테스트가 있다. 관련 테스트: tests/test_rust_dump_exporter.py:400-459.

Remediation Summary

  • FIXED MySQL column inspection now preserves textual column charset/collation in the normalized type string, preventing the reported FK mismatch class from being silently recreated as incompatible DDL.
  • FIXED Full replacement safe import now resolves a worker endpoint for the shadow database before parallel TSV load, so loader workers cannot reconnect to the original target database during shadow restore.
  • FIXED Strict dump import now rejects manifests with chunked tables that lack chunk_sha256 before target mutation. Legacy mode can still emit explicit warnings.
  • FIXED Merge import no longer blindly claims or applies recreate-style post-load DDL. Recreated targets still apply index/FK DDL; merge reports an existing-schema strategy.
  • FIXED Import success is gated by manifest row counts and writes _tunnelforge_import_report.json with verification details.
  • FIXED Python now forwards timezone_sql and strict_manifest to Rust. Rust accepts only SET SESSION time_zone for this field.
  • FIXED UI wording no longer promises automatic restoration of views, procedures, triggers, or events.

정직한 운영 복구 경로는 수정된 core로 원본 DB를 다시 Export한 뒤 새 dump를 Import하는 것이다. 기존 dump는 제한 복원 대상으로만 취급해야 하며, strict 완전 복원 산출물로 표시하면 안 된다.

Remaining Limits

Export snapshot: 이번 수습은 새 dump가 snapshot consistency를 증명하지 못하면 manifest에 snapshot_policy: not_enforced와 warning을 남기도록 했다. MySQL consistent snapshot을 실제로 공유하는 export path는 별도 후속 작업이다.

Table-level metadata: column charset/collation은 보존하지만 table engine/default collation/row format은 아직 구조화된 schema field가 아니다. 따라서 새 경로는 FK mismatch class를 줄이고 검증하지만, 모든 MySQL physical option을 완전 복제한다고 말하지 않는다.

Bottom Line

이번 수습으로 Import의 가장 위험한 경로는 검증 가능한 규칙으로 고정됐다. 다만 사용자가 제공한 기존 PROD dump는 strict 완전 복원 산출물이 아니다. 새 core로 재-export한 산출물을 표준 복구 경로로 사용해야 한다.
