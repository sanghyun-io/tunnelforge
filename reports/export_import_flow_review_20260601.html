TunnelForge Export/Import Remediation Report

Date: 2026-06-26

Scope: Rust Core dump export/import recovery work tracked from the 2026-06-01 review.

Remediation Summary

  • Python now forwards import intent fields including timezone_sql and strict_manifest to Rust Core.
  • Rust Core validates timezone_sql and only accepts a single SET SESSION time_zone statement with a safe literal value.
  • Strict import rejects incomplete dump manifests with missing chunk checksum metadata before DB connection or target mutation.
  • Classified Rust import failures such as export_invalid and post_load_validation_failed are preserved through the Python wrapper.
  • Import success is gated by per-table imported row-count verification.
  • Successful imports write _tunnelforge_import_report.json beside the dump and return the report path in the result payload.
  • Merge import no longer reapplies post-load DDL intended for recreated targets.
  • Export manifests now record snapshot_policy, strict_export, and manifest_warnings.
  • Parallel exports are marked non_consistent_parallel and non-strict instead of implying an unproven shared snapshot.
  • MySQL column inspection now preserves CHARACTER_SET_NAME and COLLATION_NAME in dump schema type metadata, and post-load FK DDL is blocked when text-column charset/collation is incompatible.
  • Shadow full replacement has been retired as a current guarantee; full replacement is documented as direct replacement against the selected target database.
  • Direct replace/recreate DDL failures now include a classified error, affected table, and operation such as drop_table or create_table.
  • Import UI wording no longer promises that every object type is restored automatically.

Residual Limits

  • Full replacement is direct replacement, not an atomic shadow-schema switch. Operators should treat replacement imports as target-mutating operations and rely on external backups for rollback.
  • View import remains best-effort. Procedures, triggers, and events still require separate review instead of automatic restoration.
  • Parallel exports are now honestly classified as non-strict, but they do not yet provide a shared consistent snapshot guarantee.
  • macOS production readiness still requires the separate real-Mac validation bundle tracked outside this report.
