Safety and Proof

TunnelForge의 다음 단계는 기능 확장이나 포지셔닝 변경이 아니다. 확인된 신뢰·데이터·실행 위험을 제거한 뒤, 하나의 폐기 가능한 DB 워크플로를 실제 사용자에게 관찰해 제품 방향을 증명하는 것이다.

결정: 다음 마일스톤은 범위를 제한한 안전성 릴리즈다. 이 릴리즈의 출구 조건을 모두 통과하기 전에는 새 엔진, 대규모 온보딩, 텔레메트리 인프라를 시작하지 않는다.
기준 브랜치 origin/main 기준 SHA 4b37bdb 최신 안정판 v2.4.0 검토 방식 8개 역할 + 레드팀 합의

1. 팀 합의

제품, UX, 아키텍처, 품질·보안·릴리즈, 시장, 운영·분석 검토를 레드팀이 반박하고 회의 의장이 최종 범위를 확정했다.

지금 고친다

SSH 서버 신원, Import 시간대 기본값, One-Click 승인 경계, DB Core 무기한 대기, 릴리즈 의존성 고정.

계약부터 쓴다

resume 식별·원자성, 취소 최종 상태, 프로토콜 호환성, 프로세스 재시작. 재설계 전에 실패 재현 테스트를 만든다.

근거 전까지 미룬다

브랜드 재포지셔닝, 원격 텔레메트리, 지원 번들, 전면 접근성 개편, 자동 롤백, 추가 DB 엔진.

핵심 원칙: DB 결과가 불확정인 상황을 자동 재시도, 빈 결과, 낙관적 성공으로 숨기지 않는다. Rust는 DB 연산을 소유하고 Python/PyQt는 UI·오케스트레이션 경계를 유지한다.

0신뢰 가능한 사용 완료 세션

릴리즈 다운로드와 clone 수는 사용자·잔존율 지표가 아니다.

확인된 강점

  • Rust Core 중심 DB 경계가 정착됐다.
  • 직접 다운로드 릴리즈·무결성 검증·익명 오류 신고 경로가 운영 중이다.
  • Python, Rust, Worker, 패키징 회귀 체계가 넓다.

해석 제한

  • 최근 릴리즈 자산 다운로드는 대체로 한 자릿수다.
  • GitHub clone 트래픽은 CI 영향을 분리할 수 없다.
  • 별 1개, fork 0개는 방향 결정에 충분한 표본이 아니다.

5. 재설계 전 실행 계약

TF-STATUS-099를 포함한 공유 프로세스·resume·취소 문제는 먼저 작은 실패 재현으로 경계를 고정한다.

Core process

  • stalled response deadline
  • mismatched response ID 명시 실패
  • 프로세스 사망 후 다음 호출 복구
  • mutation 자동 재시도 금지
  • protocol min/max compatibility

Resume / cancellation

  • source·target host/port/schema identity
  • immutable plan fingerprint
  • temp write + atomic replace
  • stale state 명시 거부
  • result/cancel 중 하나의 terminal state

제약: 각 항목은 하나의 reproducer, 하나의 계약, 그 계약을 통과시키는 가장 작은 구현으로 제한한다. 측정 없이 대규모 동시성 또는 resume 아키텍처 개편으로 확장하지 않는다.

6. Safety and Proof 릴리즈

새 기능 릴리즈가 아니라 위험 제거와 실행 경계 증명 릴리즈다.

  • 알 수 없는 SSH 호스트는 SHA-256 fingerprint 승인 전 연결하지 않고, 변경된 키를 차단한다.
  • 기본 Import가 시간대 변경 SQL을 생성하지 않는다.
  • One-Click non-dry-run은 exact plan 승인 없이는 실행되지 않거나 완전히 비활성화된다.
  • stalled DB Core 응답은 정의된 deadline 내 typed indeterminate 결과로 종료되고 손상 프로세스는 폐기된다.
  • timeout 또는 process loss 뒤 mutation은 자동 재시도되지 않는다.
  • mismatched ID와 비호환 protocol version은 명시적으로 실패한다.
  • 릴리즈 의존성은 hash lock으로 설치되며 변경 artifact는 거부된다.
  • 집중 회귀와 기존 Python·Rust·Worker 전체 게이트가 모두 통과한다.

8. 폐기 가능한 DB 워크플로 검증

원격 텔레메트리를 만들기 전에, 실제 사용자가 핵심 경로를 끝까지 완료하는지 관찰한다.

대상

유지보수자가 아닌 3–5명. 개인정보나 실제 운영 DB 대신 폐기 가능한 MySQL/SSH 환경을 사용한다.

시나리오

설치 → 연결 → fingerprint 판단 → dry-run → plan 승인 → 실행 → 결과 검증.

산출물

단계별 완료 여부, 관찰된 장애, 사용자 발화, 해결/보류 판단. 다운로드 수를 사용자 수로 환산하지 않는다.

의사결정 규칙: 3명 미만이 핵심 경로를 완료하면 새 분석 인프라나 기능을 추가하지 말고 워크플로를 더 좁히거나 수정한다.

9. 조건부 6개월 로드맵

4–6개월차는 관찰 결과가 있을 때만 활성화한다.

증거가 있으면

  • 빈도·심각도가 가장 높은 두 완료 장애 수정
  • SSH-first 가치가 반복 확인될 때만 포지셔닝 실험
  • 관찰된 차단점에 한해 첫 연결, 복구, 접근성 개선

계속 보류

  • Apple App Store 등록·유료 인증
  • 원격 텔레메트리와 별도 metrics endpoint
  • 지원 번들·SBOM 프로그램
  • 자동 롤백과 추가 DB 엔진

11. 위험 관리

잘못된 안전 개선이 새로운 데이터 위험을 만들지 않도록 응답 원칙을 고정한다.

Core timeoutDB 결과를 불확정으로 보고하고 mutation을 자동 재시도하지 않는다.
SSH TOFU알고리즘과 fingerprint를 명확히 표시하고 변경 키는 우회 없이 거부한다.
Stale plan승인을 target identity와 plan hash에 묶고 관련 변경 시 무효화한다.
범위 팽창reproducer 하나, 계약 하나, 최소 통과 구현으로 제한한다.
Hash lock 부담개발 전체가 아니라 릴리즈 입력만 고정하고 재생성·검증을 자동화한다.
작은 표본3–5회 결과는 방향성 근거로만 쓰고 product-market fit을 주장하지 않는다.
