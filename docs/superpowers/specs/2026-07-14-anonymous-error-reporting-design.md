# Anonymous Error Reporting and GitHub App Rotation Design

## Goal

Replace the distributable GitHub App credential path with an opt-in anonymous
error-reporting service that creates or updates public TunnelForge GitHub issues
without placing any GitHub credential in the desktop client. Resolve
TF-STATUS-092 through a credential-separated rolling migration that can be
disabled without affecting export, import, tunnel, database, migration, or Rust
Core operations.

## Confirmed Decisions

- The client sends HTTPS JSON to a Cloudflare Worker. It never calls the GitHub
  issue API directly and does not bundle or invoke `gh` or `curl`.
- A new dedicated GitHub App owns issue reporting. It is installed only on
  `sanghyun-io/tunnelforge` with repository Metadata read and Issues read/write.
- The reporter App private key exists only as an encrypted Worker secret.
- The CI Releaser App remains a separate credential lane unless App ID and key
  fingerprint evidence proves that it shares the exposed key.
- Reporting starts only after an affirmative user choice. No network request is
  made merely because the consent dialog was shown.
- The consent prompt appears at most twice: once initially and, after a defer,
  once more no sooner than 30 days later.
- Reports are best-effort. Reporting failure never changes the result of the
  operation that produced the error.
- Raw reports, source IP addresses, credentials, connection details, SQL, and
  database object names are not emitted to application logs or persisted by
  relay code. Cloudflare necessarily processes network metadata while serving
  the HTTPS request under its platform policies.

## Scope

This design covers:

1. A versioned, allowlisted client report schema and a defensive sanitizer.
2. An affirmative-consent dialog and matching Settings controls.
3. A Cloudflare Worker relay with independent schema validation, sanitization,
   rate limiting, idempotency, and GitHub App authentication.
4. GitHub issue creation and duplicate occurrence comments.
5. Shadow, canary, active, and off operating modes.
6. Rotation and retirement of the private key embedded in legacy releases.
7. Tests and operator evidence required before each rollout transition.

The first version does not collect general telemetry, usage analytics,
performance traces, database contents, SQL text, arbitrary attachments, or user
identity. It does not promise delivery from offline or policy-restricted
networks.

## Architecture

### Credential boundaries

The new `TunnelForge Issue Relay` GitHub App is a dedicated service identity.
Its installation is repository-scoped and its permissions are the minimum
needed to list issues, create issues, and add comments. The Worker generates a
short-lived installation token and constrains the token request to the
TunnelForge repository and the granted Issues permission.

The Worker stores the following encrypted secrets:

- `GITHUB_APP_ID`
- `GITHUB_APP_INSTALLATION_ID`
- `GITHUB_APP_PRIVATE_KEY`
- `INSTALLATION_ID_HMAC_KEY`
- `CANARY_ADMIN_TOKEN`

No secret or shared bearer credential is compiled into TunnelForge. The client
contains only the public relay URL and report schema version. A copied client
therefore provides no direct GitHub authorization.

The current `RELEASER_APP_ID` and `RELEASER_APP_PRIVATE_KEY` remain untouched
during reporter deployment. Before deleting the exposed key, the operator must
compare the old App ID and key fingerprint with the Releaser App. If they are
coupled, CI is moved to a non-exposed key and its version-bump path is tested
before deletion.

### Client components

The desktop application separates four responsibilities:

- Environment collector: reads only approved local runtime properties and
  already-known connection metadata. It performs no network discovery, WMI
  inventory, broad registry scan, new database connection, or subprocess probe.
- Report sanitizer: converts exceptions and context into the fixed schema,
  removes forbidden values, enforces limits, and fails closed on uncertainty.
- Consent policy: owns prompt count, defer time, suppression, enablement,
  disablement, and consent schema version.
- Relay transport: submits the sanitized payload over HTTPS in a retained
  background worker with bounded timeout and retry behavior.

Existing export/import dialogs continue to retain report workers until
completion. They receive only a success/failure status and public issue URL;
their primary operation result is never replaced by a reporting result.

### Versioned report schema

The initial schema is `report_schema_version = 1`. Unknown fields are rejected
by both the client serializer and the Worker validator.

| Group | Allowed fields | Constraints |
| --- | --- | --- |
| report | `report_schema_version`, `anonymous_installation_id`, `error_fingerprint` | UUIDv4 generated by TunnelForge; SHA-256 lowercase hex fingerprint |
| app | `version`, `package_kind`, `ui_language` | fixed-length strings; package kind is `source` or `frozen` |
| system | `os_family`, `os_version`, `architecture`, `locale`, `utc_offset_minutes` | enumerated family/architecture; bounded versions; offset from -840 to 840 |
| runtime | `python_version`, `qt_version`, `rust_core_version` | numeric dotted versions only |
| operation | `kind`, `db_engine`, `db_server_version`, `phase` | enumerated export/import and engine; server version reduced to major/minor |
| error | `exception_class`, `error_code`, `sanitized_message`, `app_frames` | message at most 2,000 characters; at most 20 frames |

The relay reception time is authoritative. The client does not send wall-clock
timestamps. `app_frames` contain only TunnelForge-relative module name,
function name, and source line. Absolute paths, function arguments, local
variables, and non-application frames are removed. DB server version is taken
only from metadata already available to the failed operation; the collector
does not open a new connection and omits the field when no safe value exists.

The initial schema explicitly excludes:

- person name, account name, computer name, email, and hardware identifiers;
- source IP, hostname, port, URL credentials, and connection strings;
- database, schema, table, column, index, and constraint names;
- SQL text, bound values, row data, dump data, and arbitrary user content;
- absolute/UNC paths, home directories, environment variable values, and
  command-line arguments;
- passwords, tokens, private keys, cookies, authorization headers, and DSNs;
- arbitrary issue title, body, labels, comments, and file attachments.

Adding a new field or broadening an existing field requires a schema review,
new adversarial fixtures, and a consent-version decision. Collection must not
silently expand under the existing consent.

### Sanitization pipeline

The client constructs a new dictionary from the allowlist rather than deleting
known-bad keys from arbitrary context. Structured values are normalized before
text sanitization. The sanitizer replaces credential patterns, emails,
IPv4/IPv6 addresses, URLs, DSNs, Windows/POSIX/UNC paths, and quoted database
identifiers with stable placeholders. High-entropy bearer-like strings are
removed. Output is length-bounded after replacement.

The Worker parses the body with an explicit maximum byte size, rejects unknown
content types and schema versions, validates every type/enum/length, applies an
independent sanitizer, and then reconstructs the canonical report. It never
forwards client-provided Markdown directly. If either layer cannot prove a
field safe, the field is omitted or the request is rejected.

The error fingerprint is recomputed by the Worker from normalized operation,
engine, exception class, error code, and application frame signature. A client
cannot choose the fingerprint used for issue routing.

### Consent and Settings UX

Consent states are explicit:

- `undecided`: eligible for the initial prompt.
- `deferred`: first prompt closed without activation or suppression; eligible
  again only after 30 full days.
- `prompt_exhausted`: the second prompt closed without activation; no automatic
  prompt remains.
- `suppressed`: the user checked "Do not show this again" and closed the prompt.
- `enabled`: reporting is active for the recorded consent version.
- `disabled_by_user`: a previously enabled user disabled reporting; do not
  prompt again automatically.

The prompt is shown only after the main window is ready and never over an active
database operation. It contains:

- a request to help improve TunnelForge;
- a clear statement that sanitized reports become public GitHub issues;
- expandable lists of collected and excluded data;
- the Settings path for later activation;
- an unchecked "Do not show this again" checkbox;
- a secondary "Later" action and a primary "Enable anonymous error reporting"
  action.

Window close and Escape behave exactly like "Later" and honor the checkbox.
The prompt itself performs no reporting, connectivity test, or other network
request. The second prompt is the final automatic exposure, including when the
checkbox remains unchecked.

Settings provides enable/disable, the same collection/exclusion disclosure, a
local-only payload preview built from the current allowlisted environment plus
a synthetic error, a connection test, and the last attempt status and issue
URL. A user can enable after prompt exhaustion or suppression. Material schema
expansion requires a new consent version and cannot inherit prior consent
automatically.

### Relay API and abuse controls

The Worker exposes:

- `GET /health`: reports service version and `off`, `shadow`, `canary`, or
  `active` mode without checking GitHub credentials.
- `POST /v1/reports`: accepts schema v1 reports.

The endpoint accepts only HTTPS POST with `application/json`. CORS is not used
as an authentication control. Payload size, parse depth, arrays, strings, and
request duration are bounded before GitHub access.

Rate limits combine a Cloudflare edge limit and a keyed limit derived from the
anonymous installation ID. The source IP may be used ephemerally by Cloudflare
for an edge rate-limit decision but is not written to application logs or
storage. Because both identifiers can be spoofed or shared, rate limiting is a
damage-control layer rather than authentication.

The Worker HMACs the anonymous installation ID and stores only the HMAC,
server-computed fingerprint, action result, and expiry in Cloudflare D1 for
short-lived idempotency and recurrence control. D1 stores no raw payload. A
single installation/fingerprint pair may create at most one recurrence update
within the configured window. Global creation and comment limits protect the
repository if the public endpoint is abused.

The server constructs every issue title, body, label, and comment from
validated scalar fields. Reports use the `auto-reported` label and a hidden
server-computed fingerprint marker. A matching open issue receives a bounded
recurrence comment or aggregate update; otherwise the Worker creates one issue.

### Network and firewall behavior

Environment collection is local and performs no network scan. Report delivery
uses one outbound HTTPS/443 path to the configured relay. The client opens no
listening socket and requires no firewall exception.

Corporate policy may still block the relay domain, DNS, proxy authentication,
or TLS chain. The transport honors normal proxy configuration, always verifies
TLS, uses short connect/read timeouts, and makes at most one in-memory retry.
It does not persist a report queue to disk. Offline, timeout, certificate,
proxy, 429, 4xx, and 5xx failures are recorded as bounded local status without
interrupting the source operation.

### Operating modes

- `off`: reject reports without GitHub access. Emergency rollback mode.
- `shadow`: validate, sanitize, rate-limit, and return a synthetic receipt; do
  not persist report data or access GitHub.
- `canary`: accept only an operator request authenticated with the canary admin
  token; create/update a designated synthetic issue.
- `active`: accept opted-in client reports and create/update production issues.

Mode changes are server-side and require no client release. Secrets and mode
changes use versioned Worker deployment so a prior Worker version can be
restored.

## Rolling Strategy

### Lane A: contain the exposed credential

1. Identify App ID `2735888`, its installed repositories, and exposed key
   fingerprint in GitHub Developer Settings.
2. Identify every external consumer by App ID and fingerprint.
3. Determine whether the Releaser App uses the same App and exposed key.
4. If independent, delete the exposed key immediately. If coupled, generate a
   new overlapping key, update only the Releaser secret, run a controlled
   version-bump PR, and then delete the exposed key.
5. Treat already-issued installation tokens as potentially valid until their
   one-hour lifetime expires.
6. Delete the unused repository secret `GH_APP_PRIVATE_KEY`.

Legacy `v1.13.4` through `v2.3.0` automatic reporting is intentionally allowed
to stop. This lane does not wait for the replacement feature.

### Lane B: deploy the replacement service

1. Implement and locally verify client schema/sanitizer/consent state and Worker
   validation/GitHub adapters without production secrets.
2. Create the dedicated reporter App and install it only on TunnelForge.
3. Deploy the Worker in `shadow`; run health, schema, rejection, rate-limit, and
   redaction probes.
4. Upload Worker secrets without placing them in repository files or shell
   history.
5. Move to `canary`; create one synthetic issue, submit one duplicate, verify
   issue/comment attribution and content, and close the test issue.
6. Move to `active` before publishing a client that knows the relay URL.
7. Publish the client feature through the normal protected release path.
8. Observe service counters and issue quality. Use `off` immediately if public
   content, abuse, credential, or routing behavior is unsafe.

## Error Handling

- Unsafe or unknown client field: omit locally or fail local report creation.
- Oversized, malformed, unknown-version, or forbidden server payload: return a
  stable 4xx code without echoing the payload.
- Rate limited: return 429 with bounded retry guidance; the desktop does not
  retry more than once.
- GitHub authentication failure: refresh an installation token once; then fail
  closed without logging JWT, token, PEM, or request body.
- GitHub permission/repository mismatch: fail closed and surface an operator
  health error; never broaden permissions automatically.
- Duplicate lookup failure: do not guess between create and comment. Return a
  retryable relay error to prevent issue floods.
- Client shutdown while reporting: abandon the result safely; do not block app
  shutdown or retain a report on disk.

## Observability and Retention

Allowed operational counters are request outcome, schema version, service
mode, GitHub action type, response status class, and coarse latency bucket.
Logs must not contain request bodies, source IPs, anonymous installation IDs,
HMAC values, exception messages, GitHub tokens, JWTs, or secrets.

The user-facing client stores only consent state, prompt count/time, last
attempt status/time, and last public issue URL. It does not store pending or
historical report bodies.

## Testing

- Unit tests for every allowed field, enum, bound, version normalization, and
  absent optional field.
- Adversarial sanitizer fixtures for credentials, URLs, IPv4/IPv6, emails,
  Windows/POSIX/UNC paths, DSNs, SQL, quoted object names, Unicode tricks, long
  strings, Markdown injection, and high-entropy tokens.
- Cross-language contract fixtures consumed by both Python and Worker tests so
  their schema and forbidden-value decisions cannot drift.
- Consent state tests for initial display, 29/30-day boundary, second/final
  display, suppression, enable, disable, close, Escape, and consent version
  change.
- UI tests for exact disclosure text, unchecked suppression checkbox, primary
  action, Settings path, preview, connection status, and no network on display.
- Transport tests for TLS verification, proxy behavior, timeout, one retry,
  429, offline mode, shutdown, and operation-result isolation.
- Worker tests for content type/body bounds, unknown fields, rate limits,
  server fingerprint recomputation, HMAC idempotency, issue construction,
  duplicate comments, token refresh, mode behavior, and secret-free logs.
- Local GitHub adapter tests use mocked endpoints. Production verification uses
  one designated canary issue only.
- Final repository verification includes focused Python/Worker suites, full
  Python regression, Rust Core regression, packaging smoke checks, secret scans,
  and `git diff --check`.

## Acceptance Gates

The relay may enter `active` only when:

1. Client and Worker contract fixtures both pass.
2. Forbidden-value fixtures produce no leak in the canonical issue body.
3. Shadow rejects malformed, oversized, and unknown-field payloads.
4. Canary creates exactly one issue and one bounded duplicate update.
5. The GitHub App is repository-scoped with only Metadata read and Issues
   read/write.
6. No private key, installation token, JWT, canary token, or report body appears
   in source, build artifacts, Actions artifacts, or logs.
7. `off` mode is exercised and prevents GitHub access.

The client feature may ship only after the relay is active and its emergency
off switch is verified.

## Manual Operator Actions

The repository work can implement and test all code, fixtures, documentation,
and deployment configuration. The owner must complete or authorize actions that
cross account-secret boundaries:

- inspect GitHub Developer Settings for App IDs, installations, and key
  fingerprints;
- create/install the dedicated reporter GitHub App;
- generate and upload the reporter private key directly to Worker secrets;
- authenticate Wrangler to the intended Cloudflare account;
- approve production Worker deployment and mode changes;
- delete the exposed GitHub App key after the coupling gate passes.

Each manual action will have a command/UI checklist and a verification step in
the implementation plan. The owner is not asked to paste a private key into
chat, a committed file, or a command argument.

## Deferred Work

- User-authenticated GitHub reporting or attribution.
- General product analytics and performance telemetry.
- Persistent offline report queues.
- File attachments, dumps, SQL, screenshots, and arbitrary diagnostic bundles.
- Custom paid relay domain or paid Cloudflare capacity before measured need.
- Additional report fields without schema and consent-version review.
